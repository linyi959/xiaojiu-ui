# 小九中枢 UI 设计契约

> 这个文件是“小九中枢 / Xiaojiu Command Core”的落地约束。后续真实代码必须以这里为准，不能把预览稿当灵感、把真实页面做成原版 Hermes 换皮。

## 1. 产品定位

小九中枢不是 Hermes Web UI 的美化版，而是林燚的私人 AI 中枢：

- 保留 Hermes 原有全部生产力功能
- 让小九具备稳定“在场感”
- 支撑未来多 agent、多设备、多任务调度
- 视觉上是私人 Command Room，功能上是 Hermes Pro

核心隐喻：

“林燚打开这个页面，就进入了小九的大脑控制室。”

## 2. 不可破坏原则

1. 原版 Hermes 项目不能被破坏；本项目是独立 fork。
2. 原有功能只能重组入口和视觉，不能删除。
3. 不能只换颜色；必须重构外层 Shell、信息架构和页面容器语言。
4. 中间工作区优先，左侧只做窄模块启动器，右侧 Presence 舱可响应式收起。
5. 任何“看起来做好了”的结论必须经过构建和真实页面验收。

## 3. 信息架构

长期主模块分为五组：

### Chat Core / 对话中枢
- Chat
- History
- Group Chat

目标：保留聊天能力，但让会话像“任务现场”，不是普通聊天软件。

### Missions / 任务指挥
- Monitor
- Jobs
- Usage
- Logs

目标：后台任务、巡航、运行状态、错误都进入任务雷达。

### Memory / 记忆与知识
- Memory
- Files
- History 的记忆入口

目标：让小九的记忆和知识库成为高权重区域，而不是藏在设置后面。

### Arsenal / 能力与工具
- Skills
- Models
- Terminal
- Gateways
- Channels

目标：展示小九能做什么、哪些能力可用、哪些异常。

### System / 系统驾驶舱
- Profiles
- Settings
- Provider / Gateway / Config 相关页面

目标：保留工程配置能力，但统一收进中枢视觉语言。

## 4. 主布局契约

### 左侧：Command Module Rail
- 宽度控制在 72px 左右，不做传统宽 sidebar。
- 默认只显示短标签 / glyph。
- hover/title 展示完整名称。
- 必须保留所有原始路由入口。
- 不能常驻大段介绍文案。

### 中间：Command Stage
- 是最宽区域，承载当前工作。
- 所有页面进入统一 stage surface。
- Chat 页允许独立固定输入区和消息滚动区。
- 非 Chat 页默认 stage 内滚动，避免整页破坏布局。

### 右侧：Xiaojiu Presence Dock
- 不是装饰。
- 必须显示真实状态：连接、思考、工具调用、会话、模型、活动。
- 未来可替换虚拟人物 / Live2D / 视频素材。
- 小屏时可以收窄或下沉，不能挤压主舞台。

## 5. 视觉方向

气质关键词：冷静、有生命感、私人、高级、克制、能打仗。

不要做成：
- 普通 SaaS 后台
- 原版 Hermes 换色
- 假大屏
- 赛博紫蓝模板
- 纯装饰人物页

### 色彩 token

- 背景深黑蓝：`#02040B` / `#060912` / `#0B1020`
- 主系统冷光：`#63E7FF` / `#38BDF8`
- 偏爱暖光：`#F6D58A` / `#FFB86B`
- 心跳玫瑰：`#FF6B8A`
- 错误红：`#FF4D6D`
- 成功绿：`#6EE7B7`
- 主文字：`#EAF2FF`
- 次文字：`#94A3B8`
- 弱文字：`#64748B`

## 6. 页面推进顺序

### 第一阶段：Shell 定型
验收标准：打开页面第一眼不能再像原版 Hermes。

范围：
- App 外层布局
- 左侧窄栏
- 中间 Stage
- 右侧 Presence 舱
- 顶部状态栏
- 全局背景
- 全局卡片 / 表格 / 按钮基调

### 第二阶段：Chat 页重构
范围：
- 会话列表
- 消息气泡
- thinking 区域
- tool 调用展示
- 输入框
- 空状态

### 第三阶段：核心模块重构
顺序：
1. Missions：任务雷达
2. Memory：大脑纹理
3. Arsenal：能力库

### 第四阶段：系统页统一
Settings / Logs / Usage / Monitor 等工程页统一视觉，但不优先做灵魂表达。

## 7. 交互规则

- 工具调用必须有可见行动轨迹。
- 普通视图显示“我正在做什么”，展开后才看 terminal/file/browser 细节。
- 状态动画必须有意义：idle、thinking、tool-running、error、waiting。
- 所有按钮必须有 hover/focus 状态。
- 不使用 emoji 做结构图标。
- 保持键盘可达和文本对比度。

## 8. 验收标准

每次阶段完成后必须检查：

1. `npm run build` 通过。
2. 真实页面截图或浏览器验收，不只看代码。
3. 原有路线都能打开。
4. Chat 页输入框不随整页滚走。
5. 左栏不挤压主工作区。
6. Presence 舱显示真实状态，不只是装饰。
7. 页面不出现明显原版 sidebar / 后台按钮风格回潮。

## 9. 禁止事项

- 禁止把设计契约降级成“参考”。
- 禁止为了好看删除原功能。
- 禁止把原版 UI 组件原封不动暴露在第一层产品语言里。
- 禁止未验证就说做好。
- 禁止只做表层颜色替换。
